## 1. 現状理解の要約

### 1-1. アプリの目的

筋トレユーザーが**筋肉部位ごとの疲労を記録・可視化し、回復状態から「今日どこを鍛えるか」を判断する**個人向け Web アプリ。中核価値は次の 3 点:

1. 体図で疲労状態が一目でわかる（色分け）
2. トレーニングを記録すると疲労が自動加算される（手入力の手間を減らす）
3. 時間経過で疲労が自動回復し、常に「今の状態」が見える

### 1-2. 主要機能（実装済み）

| 領域 | 機能 | 状態 |
|------|------|------|
| 認証 | Google Sign-in（Firebase Auth） | ✅ |
| 疲労マップ | 前面/背面 SVG 体図・16筋肉・4段階色分け・1分ごとクライアント側減衰再計算 | ✅ |
| 疲労入力 | 部位クリック→スライダー→`[確定]`保存（楽観的更新つき） | ✅ |
| トレーニング | 種目検索→セット/回数/重量入力→疲労自動加算（volume ベース）・デルタプレビュー | ✅ |
| 履歴 | 筋肉ごとの疲労履歴棒グラフ・回復残り時間表示・セッション履歴ページ | ✅（簡易実装） |
| リセット | 全筋肉 0% 一括リセット（確認モーダル・batch write） | ✅ |
| 運用 | シードスクリプト・CI（lint/typecheck/test/build）・Firestore rules/indexes デプロイ WF | ✅（後述の穴あり） |

### 1-3. テックスタック

Next.js 14 (App Router) / TypeScript / Tailwind CSS / Firebase Auth + Firestore / TanStack Query v5 + Zustand / Vitest + RTL + MSW（テスト 20 ファイル・約 108 ケース） / Vercel hnd1 / GitHub Actions

### 1-4. 現在の設計方針（実装から読み取れるもの）

- API Route は全て `withAuth`（Bearer IDトークン検証）経由。Admin SDK は lazy singleton
- 疲労値は追記型スナップショット（`users/{uid}/fatigueSnapshots`）。「現在値」は最新スナップショット＋線形減衰で導出
- 減衰計算はクライアント側で 1 分ごとに再実行（`useFatigueWithDecay`）。Firestore read を増やさない
- 書き込みは明示的操作時のみ（確定ボタン / 保存ボタン）。リセット・ワークアウトは batch write
- 保存 UTC / 表示 JST 固定。Vercel hnd1 + Firestore asia-northeast1 の同一リージョン
- サーバー状態 = TanStack Query、UI 状態 = Zustand の分離

この骨格は v2.0 再設計の動機（quota 超過・履歴のセッション限定）を正しく解消しており、**v3.0 でも維持すべき資産**。

### 1-5. ブランチ・リリース状態（重要な前提）

```
main   ← v1.1（Docker/GKE 時代の旧実装のまま。v2.0 は未マージ）
ver3.0 ← v2.0 の全実装が統合済み（main より 66 コミット先行）。v3.0 の統合先
ver2.0 ← 凍結（ver3.0 とほぼ同一内容）
```

**v2.0 は「実装完了・main 未リリース」の状態**。つまり v3.0 計画は「稼働中の v2.0 を壊さないよう改修する」のではなく、「未リリースの v2.0 を土台に、リリース前にどこまで v3.0 品質へ引き上げるか」という問題設定になる。破壊的変更のコストが最も低いタイミングであり、本番ユーザーデータのマイグレーション制約もほぼない（→ 要確認事項 [02_work-plan.md](./02_work-plan.md) §8）。

## 3. 重要な問題点一覧

深刻度: 高（中核価値・データ正しさを毀損）/ 中（品質・信頼性を毀損）/ 低（改善余地）。
優先度: P0（v3.0 最初期に修正）/ P1（v3.0 設計変更として対応）/ P2（UX 改善群）/ P3（運用・整理）。

### P0 — バグ・品質ゲートの穴（放置するとリリース不可能 or 誤データが蓄積）

| # | 問題 | 詳細 | 影響範囲 | 深刻度 | 放置リスク |
|---|------|------|---------|--------|-----------|
| P0-1 | **ワークアウト実施日時の初期値が UTC** | `WorkoutInputModal.tsx:26` で `new Date().toISOString().slice(0,16)` を `datetime-local` に渡している。JST ユーザーには常に 9 時間前が初期表示される | 全ワークアウト記録の既定日時 | 高 | ユーザーが気づかず保存すると記録日時が 9h ズレて蓄積。履歴の日付グルーピングも誤る |
| P0-2 | **過去日時のワークアウトが減衰なしで「今」の疲労として加算される** | `performedAt` を昨日にしても、スナップショットは `recordedAt = now`・デルタ全量で書かれる（`applyWorkoutToFatigue` は `now` 起点）。「昨晩のトレを翌朝記録」すると、実際は一晩分回復しているのに満額の疲労が表示される | 疲労値の正しさ（中核価値） | 高 | 後追い記録するほど疲労が過大評価され、アプリの判断材料としての信頼を失う |
| P0-3 | **保存・リセット失敗が無通知** | `useSaveFatigue` はロールバックのみ、`useResetFatigue` は `onError` なし。設計書 §9 の Toast 通知が未実装 | 手動保存・リセットの全失敗ケース | 中 | 「保存したつもりが消えている」を静かに起こす。楽観的更新がむしろ誤解を助長 |
| P0-4 | **CI が ver3.0 への push で走らない** | `ci.yml` の push 対象が `[main, ver2.0]`。squash merge は push イベントなので、**マージ後の ver3.0 上で CI が一度も走らない** | 統合ブランチの品質保証 | 中 | 統合後の退行を検知できないまま v3.0 開発が進む |
| P0-5 | **疲労データ取得失敗時に「読み込み中...」のまま固まる** | `useFatigueWithDecay` はエラー時 `null` を返し続け、ダッシュボードにエラー表示・リトライ手段がない | 初回表示・トークン失効時 | 中 | ユーザーには永遠のローディングに見える。401 時の再ログイン導線もない |

### P1 — 設計債務・データ整合性（v3.0 の設計変更で対応）

| # | 問題 | 詳細 | 影響範囲 | 深刻度 | 放置リスク |
|---|------|------|---------|--------|-----------|
| P1-1 | **履歴の「影響」表示が表示時点の計算式・カタログで再計算される** | `workout/history/page.tsx` はセッションの `exercises` から `computeFatigueImpact` を都度実行。疲労計算式やカタログを変えると**過去履歴の表示が遡って変わる**。v3.0 で計算式変更（強度反映）を予定しているため顕在化必至 | セッション履歴の正確性 | 高（v3.0 前提で） | 計算式変更のたびに履歴が書き換わって見え、記録アプリとしての信頼を失う |
| P1-2 | **API DTO とドメイン型の混同** | `FatigueSnapshot.recordedAt: Date` だが API 経由の実体は ISO 文字列。`fetchFatigueHistory` の戻り型は型レベルの嘘。チャート側が `Date \| string` 両対応で吸収している | 型安全性・今後の改修全般 | 中 | 型を信じたコードが実行時に壊れる。改修のたびに防御コードが増殖 |
| P1-3 | **`GET /api/fatigue/current` が毎回 16 reads** | 設計 §12-4 アプローチ A のまま。quota 削減という再設計の動機に対し、denormalized な現在値ドキュメント（アプローチ C）なら 1 read | 読み取り quota・レイテンシ | 中 | ユーザー/アクセスが増えると read 数が線形増。§19-3 のレイテンシ 150〜250ms も 16 並列 read が支配項 |
| P1-4 | **Firestore ルールがクライアント直書きを許可** | `users/{uid}/**` に本人フルアクセス。実装は全て API（Admin SDK）経由でクライアント直アクセスは 0 件（`clientDb` は未使用 export）。value 値域などの検証ルールもない | セキュリティ（攻撃面） | 中 | ブラウザコンソールから任意 shape のドキュメントを書けてしまい、API のバリデーションが無意味化する |
| P1-5 | **種目カタログ取得の `limit=20` 固定** | `useExercises` が常に limit 20。カタログが 20 件を超えると、履歴ページ・WorkoutTab の種目名解決と影響計算が静かに欠落する | カタログ拡充の障害 | 中 | v3.0 でカタログを増やした瞬間に顕在化する時限バグ |
| P1-6 | **セッション履歴のページネーション未完成** | サーバーは cursor 対応済み、UI は `limit=30` 固定で「さらに読み込む」がない | 長期利用ユーザーの履歴閲覧 | 低 | 30 件を超えた過去履歴に到達できない |

### P2 — UX（v3.0 の価値向上の中心）

| # | 問題 | 詳細 | 深刻度 | 放置リスク |
|---|------|------|--------|-----------|
| P2-1 | **モバイル未対応レイアウト** | ダッシュボードは体図+右パネルの 2 カラム固定（`min-w-[200px]` 横並び）。ジムでスマホから使うのが自然な利用文脈なのに、狭幅では操作困難 | 高 | 実利用シーンで使われないアプリになる |

| P2-6 | **ワークアウト保存が同期待ち（300〜400ms）** | §16-2 の楽観的更新は v1.3 送りのまま | 低 | 体感の重さ。ただしスピナー表示はあり許容範囲 |

## 4. 総評

v2.0 再設計の骨格（追記型スナップショット・クライアント側減衰・withAuth・quota 意識）は健全で、**アーキテクチャの土台を作り直す必要はない**。v3.0 で取り組むべきは:

1. **正しさの回復**（P0 群 + P1-1/P1-2）— 疲労値というアプリの根幹データが誤って蓄積・表示されるパスを塞ぐ
2. **利用文脈への適合**（P2-1 モバイル対応を筆頭とする UX 群）— 「ジムでスマホから使う」を成立させる
3. **中核価値の深化**（[01_spec-v3.md](./01_spec-v3.md) の新機能候補）— 「今日どこを鍛えるか」への回答をアプリ側から提示する
4. **リリースと運用の整理**（P3 群）— v2.0 実装を v3.0 として main に届けるところまでを計画に含める

*このレビューは実装コード・設定・テストを変更していない。凍結対象の設計書 3 点（function.md / design_basic.md / design_detail.md）も未変更。*
