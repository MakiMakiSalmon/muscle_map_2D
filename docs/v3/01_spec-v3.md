## 1. v3.0 の目的（プロダクト方針）

v2.0 = 「疲労を記録・可視化できる」を、v3.0 = 「**信頼できる疲労値を、実際の利用シーン（ジム・スマホ）で使える**」に引き上げる。

優先順:
1. **正しさ** — 疲労値の計算・表示・履歴が嘘をつかない（P0-1/2/3/5, P1-1/2）
2. **利用文脈** — スマホで完結する UI（P2-1/4）
3. **深化** — 「今日どこを鍛えるか」への回答提示（新機能候補）
4. **効率と防御** — read 削減・ルール強化（P1-3/4）

## 2. 仕様変更案

各項目: 変更内容 / 変更理由 / ユーザー価値 / 実装コスト / リスク / 移行方法。
コスト目安: S = 半日以下, M = 1〜2日, L = 3日以上（テスト込み・1人日換算）。

### テーマ A: ドメインの正しさ（疲労モデル）

#### A1. 過去日時ワークアウトの減衰適用（P0-2 の仕様化）【必須】

- **変更内容**: `POST /api/workout` で、疲労デルタを `performedAt` 時点で発生したものとして扱う。具体的には「デルタ自体を performedAt→now で減衰させてから現在値に加算」する:
  `nextValue = min(100, currentValue + applyDecay(delta, performedAt, muscleId, now))`
  スナップショットの `recordedAt` は従来通り `now`（履歴の並び順・監査性を維持）。`performedAt` が未来（+5分以内の許容範囲）の場合は減衰 0 として扱う。
- **理由**: 「昨晩のトレを翌朝記録」で満額加算されるのは疲労モデルとして誤り。後追い記録は最も普通の使い方。
- **ユーザー価値**: いつ記録しても疲労値が実態と一致する。記録タイミングを気にしなくてよい。
- **コスト**: S〜M（`applyWorkoutToFatigue` に performedAt 引数追加 + 単体テスト + プレビュー計算の一致確認）
- **リスク**: プレビュー（クライアント計算）とサーバー結果の乖離。→ プレビューは「デルタ +40%」表示のままなので影響は小さいが、`fatigueImpacts` レスポンスは減衰前デルタを返す仕様と明記する。
- **移行**: 保存済みデータへの影響なし（絶対値スナップショット方式のため過去データは不変）。

#### A2. スライダー初期値を減衰後の現在値に変更（P2-2、C2 の再定義）【必須】

- **変更内容**: `FatigueSlider` の初期値を `entry.savedValue` → `entry.currentValue` に変更。remount キーも `${muscleId}-${entry.currentValue}` ではなく `muscleId` + 保存完了トリガーにする（減衰 tick でスライダー操作中に remount されない設計とセットで）。
- **理由**: パネルの表示値（currentValue）とスライダー位置が一致しないのは、v2.0 で C2 として直したはずの「初期値バグ」の変種。ユーザーのメンタルモデルは常に「今の値」。
- **ユーザー価値**: 表示とスライダーが一致し、誤確定（古い高値のまま保存して疲労が跳ね上がる）が消える。
- **コスト**: S
- **リスク**: 操作中に 1 分 tick で currentValue が変わると draft が意図せずリセットされ得る → remount 条件を「筋肉変更 or 保存完了」に限定して回避（テストで担保）。
- **移行**: なし（表示ロジックのみ）。

#### A3. 疲労インパクトのセッション永続化（P1-1）【必須】

- **変更内容**: `POST /api/workout` 保存時に、計算した `fatigueImpacts`（MuscleId→デルタ）をセッションドキュメントに保存する。履歴表示は保存値を優先し、フィールドがない旧セッションのみ現行のクライアント再計算にフォールバック。
- **理由**: 計算式を今後変更（A4）しても過去履歴の表示が変わらないようにする。記録アプリの原則は「記録した時点の事実を保存する」。
- **ユーザー価値**: 履歴が改変されない安心感。計算式改善を躊躇なく行える土台。
- **コスト**: S（API 1 行 + 履歴ページのフォールバック分岐 + テスト）
- **リスク**: ほぼなし。ドキュメントサイズ増は僅少（最大 16 エントリの map）。
- **移行**: 後方互換（旧セッションはフォールバック表示）。マイグレーション不要。

#### A4. 強度（RPE）を疲労計算に反映【採用確定（Q3）】

- **変更内容**（推奨案）: RPE（主観的強度 1〜10、任意入力）を種目行に追加し、係数化する:
  `intensityFactor = 0.6 + RPE × 0.06`（RPE 未入力は 1.0 = 現行互換、RPE 10 で 1.2 倍）。体重ベース案（design_detail §4-2 (a)）はユーザープロフィール（体重）の新規管理が必要になるため採用しない。
- **理由**: 現行は volume（sets×reps）のみで「5×5 高重量 < 3×10 軽量」となり生理学的に逆転する（v2.0 で既知の妥協）。
- **ユーザー価値**: 高強度低回数トレの疲労が正しく反映される。
- **コスト**: M（型・API・UI・プレビュー・テスト。A3 が先行必須 — 式変更で履歴表示が変わらないため）
- **リスク**: 入力項目が増えて記録が面倒になる → 任意入力・デフォルト互換で緩和。係数の妥当性は運用しながら調整（純粋関数差し替え可能な構造は既にある）。
- **移行**: `WorkoutExerciseInput` に `rpe: number | null` を追加（旧データは null = 係数 1.0）。後方互換。

#### A5. `head` のラベルを「首」に変更【採用確定（Q8）】

- **変更内容**: `MUSCLE_LABELS.head` / `MUSCLE_GROUP_LABELS.head` を「頭部」→「首」に変更。`MuscleId` の `head` は不変（ID 改名は Firestore データ・SVG・カタログに波及するため実施しない）。
- **理由**: 対応種目（ネックエクステンション）が示すとおり実体は首。体図クリック対象としても「首」が正確。
- **コスト**: S（定数 2 箇所 + ラベル文字列を検証するテストの追随）
- **リスク・移行**: なし（表示文字列のみ）。

#### B1. モバイルファーストレイアウト（P2-1)【必須】

- **変更内容**: ダッシュボードをブレークポイントで切り替える。
  - モバイル（< md）: 体図を上部に大きく表示、パネルは下部シート（タブ）に縦積み。ボトムバーはグループバッジの横スクロール。モーダルは全画面シートに。
  - デスクトップ（≥ md）: 現行 2 カラム維持。
  - あわせて Modal の focus trap（P2-5）を実装。
- **理由**: 利用シーンの中心はジム×スマホ。現行はデスクトップ前提。
- **ユーザー価値**: 実際に使える。中核価値の前提条件。
- **コスト**: L（レイアウト再構成 + コンポーネントテスト + 実機確認）
- **リスク**: 大きめの UI 差分。ロジック（hooks 層）に触れないよう、レイアウト層のみの変更に限定する。
- **移行**: なし。

#### B2. エラーフィードバック基盤（P0-3/P0-5）【必須】

- **変更内容**:
  - 軽量 Toast コンポーネント（`components/ui/Toast.tsx` + Zustand の通知キュー）を自作（外部パッケージ追加なし）
  - `useSaveFatigue`/`useResetFatigue`/`useWorkout` の `onError` で Toast 通知
  - ダッシュボードに fatigue current のエラー状態 UI（メッセージ + 再試行ボタン、401 時は `/login` へ誘導)
- **理由**: 設計書 §9 が定めた挙動が未実装。楽観的更新 + 無通知ロールバックは「保存できたと誤解させる」最悪の組み合わせ。
- **ユーザー価値**: 失敗がわかる・やり直せる。データ喪失の不安がない。
- **コスト**: M
- **リスク**: 低。
- **移行**: なし。

#### B3. 履歴チャートの時間軸化 + 回復曲線（P2-3・G6 の再設計）

- **変更内容**: 履歴チャートを「実時間軸の折れ線/ステップチャート + 現在から完全回復までの予測破線」に変更。SVG 自作（チャートライブラリは追加しない）。期間は直近 7 日固定から開始。
- **理由**: 等間隔棒グラフは記録間隔と回復傾向を消してしまい、「分析」の役に立たない。design_basic の意図（回復曲線オーバーレイ）を実装可能な形に再定義。
- **ユーザー価値**: 「この筋肉はいつ回復するか」「最近負荷をかけ過ぎか」が読み取れる。
- **コスト**: M〜L
- **リスク**: SVG チャートの実装量。まず 1 筋肉表示に限定して抑える。
- **移行**: なし（表示のみ）。

#### B4. ワークアウト保存の楽観的更新（§16-2 の実施）

- **変更内容**: design_detail §16-2 のパターンをそのまま実装（クライアント側で impacts を予測計算し即時反映、失敗時ロールバック）。A1 導入後は予測計算にも performedAt 減衰を適用して一致を保つ。
- **理由**: v2.0 で「サーバーとの計算一致を確認してから」と延期されていた項目。A3 でサーバー計算結果が保存されるため、一致検証もしやすくなる。
- **ユーザー価値**: 保存の体感 0ms。モーダルが即閉じる。
- **コスト**: S〜M
- **リスク**: クライアント/サーバー計算の乖離 → 共通の純粋関数を使用しているため低い。`onSettled` の invalidate で必ず収束。
- **移行**: なし。

#### B5. 「今日のおすすめ」表示（回復状態からの提案）【採用確定（Q4）・新機能】

- **変更内容**: ダッシュボード上部に「回復済み（currentValue < 30%）の筋肉グループ」をバッジ表示し、「今日鍛えられる部位」を提示する。既存の `useFatigueWithDecay` の派生値のみで実装（新規 API・新規データなし）。
- **理由**: アプリの中核目的「今日どこを鍛えるか」への回答を、色を読み解かせるのではなくアプリ側から言語化する。
- **ユーザー価値**: 開いた瞬間に意思決定が終わる。アプリを開く動機になる。
- **コスト**: S〜M
- **リスク**: 「おすすめ」の閾値設計がナイーブ（30% 固定）→ v3.0 では説明可能な単純ルールに留め、係数調整は運用で。
- **移行**: なし。

#### B6. ワークアウトの削除【採用確定（Q5）・削除のみ。編集は v3.1 以降】

- **変更内容**: `DELETE /api/workout/{id}` と履歴ページからの削除 UI（確認モーダルつき）。削除時は該当セッション由来のスナップショットも削除し、影響筋肉の現在値を再導出（直前スナップショットが自然に最新になる追記型の利点を活用）。**編集（PATCH）は実装しない** — 誤入力は「削除 + 再入力」で代替する。
- **理由**: 誤入力の救済が「手動上書き or 全リセット」しかないのは、記録アプリとして不便が大きい。編集は削除より整合性設計が重く、削除+再入力で同じ結果を得られるため見送り。
- **ユーザー価値**: 誤記録を恐れず入力できる。
- **コスト**: M〜L（削除の整合性設計・テスト）
- **リスク**: スナップショット削除の整合性（同一筋肉に後続の手動保存がある場合の扱い等）→ 「セッション削除 = 該当スナップショットの削除のみ。後続スナップショットには触れない」と割り切る（[design.md](./design.md) D3 参照）。
- **移行**: なし（新規 API）。

### テーマ C: データ・パフォーマンス・セキュリティ

#### C1. 現在疲労値の単一ドキュメント化（P1-3、§12-4 アプローチ C の精緻化）

- **変更内容**: `users/{uid}/state/fatigueCurrent` という**単一ドキュメント**（16 筋肉分の map フィールド）を追加し、スナップショット書き込みと同一 batch で merge 更新。`GET /api/fatigue/current` はこの 1 ドキュメント読み取りに変更。`fatigueSnapshots` は履歴専用になる。
- **設計注記（筋肉ごとのサブコレクション案を却下した理由）**: Firestore の read 課金は「クエリ返却ドキュメント数」単位。`fatigueCurrent/{muscleId}` の 16 doc 構成では 1 クエリでも 16 reads のままで削減にならない。単一 doc なら真に 1 read。サイズは 16 筋肉 × 約 100B で 1MB 制限に対し余裕、書き込み頻度も個人利用で単一 doc の秒間 1 write 目安に届かない。
- **理由**: v2.0 設計自身が「読取コストが問題になれば C へ」と予告していた改善。`GET /api/fatigue/current` が 16 reads → 1 read、`POST /api/workout` 内の現在値取得も 16 クエリ → 1 read になる。
- **ユーザー価値**: 表示・保存の高速化（16 並列クエリの RTT が消える）。将来のアクセス増への耐性。
- **コスト**: M（読み書き 4 箇所 + テスト）
- **リスク**: 二重管理（snapshot と current の不整合）→ 同一 batch 書き込みで原子性を担保。読み側は current を単一ソースとする。
- **移行**: **不要（Q2: 本番データなし）**。ドキュメント欠落時は全 16 筋肉デフォルト値（value 0）を返し、初回書き込みで自然に生成される。lazy migration は実装しない。

#### C2. Firestore ルール強化（P1-4）

- **変更内容**: `users/{uid}/**` のクライアント直接アクセスを read/write とも遮断（全て API 経由のため）。`exercises` は現行維持（認証済み read のみ）。※運用に影響する設定変更のため、専用ブランチ・専用 PR とし、デプロイタイミングを明示する。
- **理由**: 使っていない直接アクセス経路を開けておく理由がない。API のバリデーションを唯一の入口にする。
- **ユーザー価値**: 直接は見えないが、データ破壊・偽装への防御。
- **コスト**: S（ルール変更 + 手動デプロイ + 動作確認）
- **リスク**: 将来クライアント直読みを使う機能（リアルタイムリスナー等）を作る場合は再度開ける必要がある — v3.0 スコープにはないので問題なし。
- **移行**: `firebase deploy --only firestore:rules`。ロールバックはルールを戻すだけ。

#### C3. 種目カタログの拡充と取得仕様の整理（P1-5）

- **変更内容**: ① `useExercises` / `GET /api/exercises` に「全件取得モード」（`limit` 上限を撤廃 or カタログ件数上限を仕様化: 例 200 件）を定義し、種目名解決用途は全件キャッシュ（staleTime 24h は現行踏襲）を使う。② シードデータを主要種目 30〜40 件程度へ拡充（プル系・レッグ系・体幹のバリエーション）。
- **理由**: 現行は 20 件を超えた瞬間に履歴の名前解決が壊れる時限バグ。カタログ拡充は記録機能の実用性に直結。
- **ユーザー価値**: 自分の種目が見つかる。検索の意味が出る。
- **コスト**: S（API/フック） + M（カタログ作成・筋肉マッピングの妥当性確認）
- **リスク**: カタログ品質（マッピングの妥当性）はレビューが必要。
- **移行**: シード再実行（upsert なので安全）。

### テーマ D: 品質・運用

#### D1. CI/デプロイ運用の修正（P0-4・P3-2）【必須・最優先】

- **変更内容**: ① `ci.yml` の push 対象に `ver3.0` を追加。② `deploy.yml` は main 限定を維持し、開発中の rules/indexes 変更は「PR マージ後に手動デプロイ」と運用ルールに明文化（CLAUDE.md v3.0 版へ記載）。
- **コスト**: S / **リスク**: なし / **移行**: なし。

#### D2. API DTO 型の分離（P1-2）

- **変更内容**: `src/types/api.ts`（または zod スキーマからの `z.infer`）で API レスポンス型（`recordedAt: string` 等）をドメイン型と分離。`fetch` 層の戻り型を DTO に修正し、`Date | string` の防御コードを削減。
- **ユーザー価値**: 直接なし（改修速度と品質の土台）。
- **コスト**: M / **リスク**: 広く浅い差分。テストが既にあるため安全に進められる。

#### D3. E2E スモークテスト【採用確定（Q6）・パッケージ追加は実装ブランチで実施】

- **変更内容**: Playwright で「ログイン（モック or テストユーザー）→ スライダー保存 → 体図色変化 → ワークアウト記録 → 履歴表示」の 1 本道スモーク。CI はスモークのみ実行。
- **理由**: 統合時の退行検知が unit/component だけでは弱い（P3-3）。
- **コスト**: L（環境構築込み）/ **リスク**: Firebase Auth の E2E は手間 → 認証モック方針（エミュレータ or カスタムトークン）を実装前に PoC で確定する（[design.md](./design.md) D7）。パッケージ追加（`@playwright/test`）は `test/e2e-smoke` ブランチ内で実施。

#### D4. ドキュメント v3.0 化【必須】

- **変更内容**: `docs/v3/function.md`（v3.0 機能一覧・確定版）・`docs/v3/design.md`（v2.0 設計書との差分形式）を新規作成し、README / CLAUDE.md / AGENTS.md を v3.0 運用に更新。v2.0 の設計書 3 点は「凍結・歴史的記録」の注記を冒頭に追記するのみ。
- **コスト**: M / **リスク**: なし。

## 3. 破壊的変更の候補と移行

「破壊的」= 保存済みデータ・API 契約・ユーザーの既存操作感を変えるもの。**v2.0 が main 未リリースである今が、破壊的変更の最安タイミング**。

| # | 変更 | 何が破壊的か | 必要な理由 | 影響 | 移行手順 | 後方互換 |
|---|------|-------------|-----------|------|---------|---------|
| BC1 | C1: `state/fatigueCurrent` 単一ドキュメント追加 | スキーマ追加（読み経路の変更）。厳密には追加的 | read 16→1 | API 実装・rules | 移行不要（Q2: 本番データなし）。欠落時はデフォルト値、初回書き込みで生成 | 維持する（snapshots は不変） |
| BC2 | A3/A4: `workoutSessions` に `fatigueImpacts`・`rpe` フィールド追加 | 旧ドキュメントにフィールドなし | 履歴の不変性・強度反映 | 履歴表示・API | フィールド欠落時のフォールバック表示 | 維持する（フォールバックで吸収） |
| BC3 | A1: ワークアウト加算式の変更 | 同じ入力に対する疲労値が変わる（操作感の変化） | 疲労モデルの正しさ | POST /api/workout・プレビュー | 過去データは絶対値保存のため不変。リリースノートに挙動変更を明記 | 過去データは互換。挙動は非互換で良い（正しさ優先） |
| BC4 | C2: Firestore ルール遮断 | クライアント直アクセスが不可になる | 攻撃面の最小化 | 実利用コードに影響なし（直アクセス 0 件） | rules デプロイのみ。問題時は旧ルールに戻す | 維持不要 |
| BC5 | A2: スライダー初期値変更 | C2（v2.0 機能要件）の文言と異なる挙動 | 表示との一貫性 | UI のみ | v3.0 機能一覧で C2 を再定義 | 不要 |
| BC6 | `head` の扱い → **決定（Q8）: ラベルのみ「頭部」→「首」に変更** | ID 改名は全データ・SVG・カタログに波及するため実施しない | ドメイン用語の正確さ | `MUSCLE_LABELS` / `MUSCLE_GROUP_LABELS` の 2 箇所のみ | なし（表示文字列のみ） | ID `head` は維持 |

**API 契約**: v3.0 では既存エンドポイントのレスポンス形状を変えない（追加のみ）。クライアントとサーバーは同時デプロイされる構成（同一 Next.js アプリ）のため、API バージョニングは導入しない。

## 4. 推奨アーキテクチャ方針

### 維持すべき設計（v2.0 の資産）

| 設計 | 判断理由 |
|------|---------|
| Next.js 14 App Router + API Route + Firebase 構成 | 動作しており、個人アプリの規模に対して十分。刷新の便益がコストに見合わない |
| 追記型スナップショット + クライアント側減衰再計算（§12-5） | quota 対策の核。1 分 tick の再計算は read 0 で「生きている体図」を実現しており、v3.0 の全機能がこの上に乗る |
| `withAuth` + Admin SDK lazy singleton | 認証の単一路。§19 のレイテンシ設計も有効 |
| 確定ボタン保存（debounce なし） | 書き込み quota と誤操作防止の両立。変更理由なし |
| TanStack Query（サーバー状態）+ Zustand（UI 状態）の分離 | 責務が明確で、テストも書けている |
| UTC 保存 / JST 表示・hnd1 + asia-northeast1 | 変更理由なし |
| Vitest + RTL + MSW のテスト基盤 | 20 ファイル・約 108 ケースの資産。v3.0 の変更はすべてこの上でテストを追加する |

### 変更すべき設計

| 設計 | 変更方向 | 理由 |
|------|---------|------|
| 現在値の導出（16 並列 read） | `fatigueCurrent` denormalization（C1） | v2.0 設計書自身が予告していた進化。read 16→1 |
| ドメイン型 = API 型の兼用 | DTO 分離（D2） | `recordedAt: Date` の型の嘘を解消 |
| エラー処理「設計書にあるが未実装」 | Toast + エラー状態 UI（B2） | §9 を実装で満たす |
| Firestore ルール（本人フルアクセス） | クライアント直アクセス遮断（C2） | 使わない入口は閉じる |
| ワークアウト加算の時間意味論 | performedAt 起点の減衰（A1） | ドメインの正しさ |
| CI 対象ブランチ | ver3.0 追加（D1） | ブランチ戦略との整合 |

### 採用しないほうがよい設計（検討済み・却下）

| 案 | 却下理由 |
|----|---------|
| Supabase / PostgreSQL への DB 移行（旧 G1） | quota 問題は書き込み設計 + C1 で解決する。移行コスト（全データ層・認証連携・rules 相当の再構築）に見合う便益がない |
| Next.js 15/16 への更新・React 19 化 | 動作中の 14 系で v3.0 の全要件が満たせる。フレームワーク更新は v3.0 の価値（正しさ・UX）に寄与せず、リスクだけ持ち込む。セキュリティパッチ（14.2.x 内のパッチ更新）のみ追随 |
| Edge Runtime / Middleware 認証 | Admin SDK 非対応（v2.0 で結論済み）。変更なし |
| RSC 化・サーバーコンポーネントでのデータフェッチへの再設計 | 認証がクライアント保持の IDトークン前提。CSR + API の現構成で体感速度の問題は C1/B4 で解決する |
| チャートライブラリ（recharts 等）導入 | B3 は SVG 自作で足りる。依存追加はバンドルサイズと保守の負債 |
| リアルタイムリスナー（onSnapshot）による同期 | read quota を能動的に増やす方向であり、v2.0 再設計の動機に逆行。減衰はクライアント計算で足りる |
| debounce 自動保存への回帰 | v2.0 の結論（§12-2）を維持 |
